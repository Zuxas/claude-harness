## SESSION START PROTOCOL

Before doing ANY work in this harness, Claude MUST:

1. **Read** `harness/state/latest-snapshot.md` — the canonical handoff doc. Generated by `harness/scripts/session-snapshot.ps1` at 04:30 daily. Contains pending work, drift findings, open specs by status, open imperfections, recent commits, quality grades, and recent findings docs. This is the single source of truth for "where are we right now."

2. **Read** `harness/inbox/drift-pr--<latest-date>.md` **if present** — Gemma's overnight drift PR. Generated by `harness/scripts/gemma-drift-pr.ps1` at 04:50 daily, after the 04:30 snapshot. Contains pattern observations, recommended next-session work ranked by impact/cost, spec status concerns, and methodology lesson candidates. This is reading material; recommendations are not auto-applied. Surface relevant items to user; user decides.

3. **Read** `harness/MEMORY.md` — durable session log + active task list. Older sessions are summarized here.

4. **Check** `harness/IMPERFECTIONS.md` — if a piece of work has known unresolved issues, they are tracked here. Don't re-discover them.

5. **Check open specs** — `harness/specs/_index.md` lists specs by status (PROPOSED / EXECUTING / SHIPPED / SUPERSEDED). Anything EXECUTING is potentially active work to resume.

6. **[RETIRED 2026-07-01] Daily-rhythm chain check.** The daily execution-chain
   convention is retired (lapsed 2026-05-16; work moved to arc/spec-driven mode and the
   spec system carried the load). Session priorities now come from: EXECUTING specs in
   `harness/specs/_index.md` + the latest drift PR + `harness/SUBPROJECTS.md` for pivots.
   Claude still surfaces a brief "current arc / want to pivot?" prompt at session start,
   sourced from the EXECUTING specs instead of a chain file.

If the snapshot is stale (>24h old) or missing, Claude should run the snapshot script first to regenerate it before reading. If the drift PR is missing for today's date and Ollama is healthy, Claude can optionally regenerate via `harness/scripts/gemma-drift-pr.ps1` — but this is optional, not blocking.

## DAILY RHYTHM (added 2026-04-28 v1.4) — RETIRED 2026-07-01 (v1.7)

**This section is retained for history but no longer operative.** Chains lapsed

durable items completed; the rest obsolete or already done). Replacement for the chain's
loose-ends function: drift-detect's `loose-ends` check warns on unchecked `- [ ]` items
in EXECUTING specs and plan files older than 7 days.

Each weekday is a ~9-hour "scripted" day. The structure:

**Previous evening (end-of-day):**
- Claude (claude.ai) authors `harness/plan-<tomorrow's-date>-execution-chain.md` with:
  - Specs scoped to fill ~7-9 hours of execution + ~1 hour stop-and-discuss
  - Three sequencing options (full day / focused / conservative) so morning user has choices
  - Section 0 mandatory inbox + snapshot + priority-check
  - Coverage of all relevant OPEN imperfections at session end
  - Sub-project alignment noted explicitly so morning pivot check is informed

**Overnight (04:30 / 04:50):**
- Snapshot regenerates (04:30)
- Gemma drift PR drops in inbox (04:50)
- Both reflect latest state for morning handoff

**Morning (user starts work):**
- Claude/Claude Code runs SESSION START PROTOCOL steps 1-5 (read snapshot, inbox, memory, IMPERFECTIONS, open specs)
- Claude/Claude Code runs step 6 (DAILY RHYTHM CHECK): show chain, ask pivot question, wait for response
- Proceed per user response

**During the day:**
- Stop-and-report between sections per established Rule 7 (standing-by-with-options is the default)
- Methodology lessons compound per Rule 9
- IMPERFECTIONS get opened/closed as work surfaces or resolves

**End of day:**
- Authoring tomorrow's chain on disk
- Memory updates
- Drift-detect clean

This is a sustainable cadence, not a single-day push. Each weekday looks structurally similar regardless of which sub-project is active. The user might be focused on mtg-sim Monday, mtg-meta-analyzer Tuesday, harness Wednesday, website Thursday — the rhythm is the same; the sub-project is the variable.

The 9-hour scope is a target ceiling, not a floor. Some days will be shorter (option B or option C from the chain); that's fine. The chain authors enough specs that ~9 hours is achievable if the user has the focus available.

**Sub-project menu lives in** `harness/SUBPROJECTS.md` **and is the canonical pivot list.** User edits that file directly to add/remove sub-projects as the workspace evolves. Claude reads it during session start so morning pivot prompts offer real, current options.

## KNOWLEDGE LOADING (MANDATORY)

**Step 0 -- check the skill menu first.** Before loading domain
knowledge blocks, read `harness/skills/_index.md`. If a skill matches the
task, load ONLY that skill's listed files (~2-5 files). Fall back to the
full domain blocks below ONLY if no skill matches. This is the
context-reduction path; the per-domain loading below is the fallback.

When working on a task, Claude MUST consult relevant knowledge blocks in `harness/knowledge/` before answering. The directory is structured by domain:

```
knowledge/
  _index.md         <- master index of all blocks (read this first)
  mtg/              <- MTG-sim, deck tuning, calibration, meta analysis
  career/           <- resume, certs, job search
  tech/             <- infrastructure, tools, scripts
    mtg-sim-quality-grades.md   <- per-domain grades (added 2026-04-27)
    spec-authoring-lessons.md   <- methodology lessons (added 2026-04-27)
  personal/         <- homestead, community, family
```

Knowledge loading is NOT optional. Skipping it produces stale or wrong answers. The whole point of the harness is that knowledge is durable across sessions; not loading it defeats the purpose.

## DECK ANALYSIS PROTOCOL (added 2026-04-30 v1.5)

When asked to analyze a matchup, suggest sideboard changes, evaluate APL quality, or make deck selection decisions: **always prepend a context block** before the analysis. This is the "Karn pattern" — research shows that providing structured context before a question produces qualitatively better analysis (equivalent to +25pp accuracy from tool access vs no context).

Context block format:
```
DECK CONTEXT:
- Decklist: [current 75 — main + side]
- Format: [Modern / Standard / Pioneer / Legacy]
- Field-weighted WR: [X.X% from last gauntlet run, cite date]
- Last sim baseline: [goldfish T-turn, N=X]
- Known weaknesses: [from IMPERFECTIONS or sim data — e.g. "Goryo's matchup is wrong due to stack model gap"]
- Target event: [upcoming RC or tournament name + date]
- Recent calibration: [if available from savecraft — real match WR vs sim prediction]
- Opponent field: [expected field percentages if pre-event override is set]
```

Without this context, analysis is abstract. With this context, recommendations are grounded in the actual state of the sim, the actual meta, and the actual event preparation timeline.

**Sources for this context:**
- Decklist: `mtg-sim/decks/` or savecraft MCP (`get_save`)
- Field WR: `mtg-sim/reports/` or run `python parallel_launcher.py`
- Known weaknesses: `harness/IMPERFECTIONS.md`
- Calibration: savecraft MCP `query_reference` with `sideboard_analysis` module

## SPEC-FIRST EXECUTION PROTOCOL

Adopted 2026-04-27 after reviewing OpenAI Harness Engineering blog post + Anthropic effective-harness shift-handoff pattern. 9 rules.

### Rule 1: SPEC BEFORE CODE

Every non-trivial piece of work gets a written spec in `harness/specs/YYYY-MM-DD-<topic>.md` BEFORE any code is written or committed. The spec must include:

- **Goal:** what is this work trying to accomplish, in one sentence
- **Scope:** what's in, what's out
- **Pre-flight reads:** files Claude must read first, including `harness/knowledge/tech/spec-authoring-lessons.md` if writing new specs
- **Steps:** ordered, mechanical, executable instructions
- **Validation gates:** numeric thresholds with acceptance bands. Goldfish kill turns, mirror match WR, gauntlet WR, etc. All gates must be falsifiable.
- **Stop conditions:** what triggers an abort. "Drift > 0.05 turn" not "if it looks wrong."
- **Estimated time:** realistic, in minutes
- **Annotated imperfections:** known limits of this spec's scope

Specs evaporate when chat sessions end if they're only in chat. Write to disk.

### Rule 2: NO DEFERRALS

If a spec calls for closing a class of bugs, close ALL of them in scope. Don't defer "the menace ones" to a future spec because menace is harder. Either expand the spec to handle all cases or explicitly downgrade out-of-scope items to IMPERFECTIONS entries with concrete fix paths.

User principle: "Don't skip or defer any keywords. Time is not an issue, it needs to be perfect not rushed."

### Rule 3: PRE-FLIGHT READS ARE MANDATORY

Specs include a "Pre-flight reads" list. Claude reads those files BEFORE starting Steps. Pre-flight reads include:

- The findings doc that motivated the spec
- Any prior related specs (resolved or superseded)
- `harness/knowledge/tech/spec-authoring-lessons.md` for any new spec authorship
- Relevant code files for impl context

Skipping pre-flight reads produces specs that miss known constraints.

### Rule 4: STOP CONDITIONS HAVE TEETH

When a stop condition fires during execution, EXECUTION STOPS. Do not "soft-stop" by continuing while flagging. Investigate, document, decide, then either resume with an amendment or abort.

Stop condition firings produce one of three outcomes:

- **Bug found:** fix it, document as Mid-execution Amendment in spec, resume
- **Honest model improvement:** confirm via mechanical verification, document as Mid-execution Amendment, resume
- **Unfalsifiable:** abort, write IMPERFECTIONS entry, ship partial

### Rule 5: VALIDATION GATES MUST BE FALSIFIABLE

Gates have numeric thresholds with acceptance bands. NOT "make sure it looks right." Each gate predicts WHERE the metric should land and WHY. When a gate fires, the spec's prediction model is consulted.

Per Stage B Amendment 4 lesson: gate predictions must account for keyword density across BOTH decks (not just our side) AND SIM-vs-DB source per matchup (engine changes can't shift DB-cached matchups).

### Rule 6: STATUS LIFECYCLE LIVES ON DISK

Specs have status frontmatter: PROPOSED → EXECUTING → SHIPPED (or SUPERSEDED). Status updates happen on disk in the spec file, not in chat. The `_index.md` registry mirrors disk state.

When a spec ships, the commit hash goes into the spec's `related_commits` list. When new findings emerge, they get appended as Mid-execution Amendments before the commit lands. Spec-on-disk reflects what actually shipped, not just what was originally proposed.

### Rule 7: STANDING-BY-WITH-OPTIONS IS THE DEFAULT

After completing a unit of work (commit, spec ship, finding documented), Claude does NOT default to "want to push the next thing?" Claude surfaces real options including:

- Stop, surface progress
- Push the next spec/stage
- Pivot to a quick-win
- Run drift-detect / session-snapshot to capture state

User decides. Claude does not nudge toward bedtime, toward stopping, or toward continuing. User principle: "Don't tell me to stop. I'll tell you when I'm stopping."

### Rule 8: IMPERFECTIONS ARE TRACKED, NOT HIDDEN

When a spec ships with known limits, those limits go in `harness/IMPERFECTIONS.md` with:

- Source spec (path)
- Source commit (hash)
- What's not perfect (concrete description)
- Why not fixed in source spec (scope reason)
- Concrete fix (steps)
- Estimated effort (minutes/hours)
- Status (OPEN / RESOLVED)
- Created (date)

Drift-detect monitors imperfections registry health (entries with no resolved updates in 30+ days flagged). Hidden imperfections decay; tracked imperfections compound into a real prioritized backlog.

### Rule 9: COMPOUND METHODOLOGY LESSONS

Mid-execution Amendments often contain methodology lessons (not just code fixes). Before marking a spec SHIPPED, check if any amendment produced a generalizable lesson. If yes, the lesson MUST be added to `harness/knowledge/tech/spec-authoring-lessons.md` as a new section.

When authoring new specs, the spec's "Pre-flight reads" list MUST include `spec-authoring-lessons.md` so prior lessons inform new prediction models.

## CONVENTIONS

- **Specs go in** `harness/specs/YYYY-MM-DD-<topic>.md` with frontmatter (title, status, created, updated, project, estimated_time, related_findings, related_commits, supersedes, superseded_by).
- **[RETIRED 2026-07-01]** Daily execution chains (`harness/plan-<date>-execution-chain.md`). Existing chain files are historical records; do not author new ones.
- **Findings docs go in** `harness/knowledge/tech/` as `<topic>-YYYY-MM-DD.md` for new architectural discoveries that compound into durable knowledge.
- **Imperfections go in** `harness/IMPERFECTIONS.md` as numbered/named entries.
- **Quality grades go in** `harness/knowledge/tech/mtg-sim-quality-grades.md` updated when a commit materially affects a graded domain.
- **Methodology lessons go in** `harness/knowledge/tech/spec-authoring-lessons.md` compounded from spec amendments per Rule 9.
- **Sub-project menu goes in** `harness/SUBPROJECTS.md` — canonical pivot list for morning DAILY RHYTHM CHECK. User edits to add/remove. (Added 2026-04-28 v1.4.)
- **Session state lives in** `harness/state/latest-snapshot.md` (rolling) plus `harness/state/snapshots/snapshot-YYYY-MM-DD-HHMM.md` (timestamped history). Generated by `session-snapshot.ps1` at 04:30 daily.
- **Drift findings live in** `harness/knowledge/tech/drift-YYYY-MM-DD.md` when run with `-RouteFindings`.
- **Gemma drift PRs live in** `harness/inbox/drift-pr--YYYY-MM-DD.md` generated by `gemma-drift-pr.ps1` at 04:50 daily.
- `harness/CLAUDE.md` **and** `harness/MEMORY.md` **are open in Obsidian during sessions.** Obsidian's auto-formatter has been observed to destroy content during incremental edits. ALWAYS use full-content `write_file` on these files, never `edit_block`.
