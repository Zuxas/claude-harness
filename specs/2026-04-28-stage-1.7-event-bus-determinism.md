## Summary

`mtg-sim` has third-source mutation that breaks determinism across sequential games with the same seed. Stages 1.5 and 1.6 closed two known mutation sources (Card state leakage, APL state leakage). This spec finds and closes the third via diagnostic-first instrumentation: hash logging at game start/end of all candidate mutable state, run 5 games seed=42 sequentially, diff hashes, locate divergence.

Once shipped, gauntlet runs can use within-matchup parallelism (run multiple games concurrently within a single matchup) without contaminating each other, which reduces gauntlet wall time by 3-5x.

## Background

`mtg-sim` aims for determinism: same seed → same game outcome, regardless of run order or parallelism. Empirically, 5 sequential games with seed=42 produce different outcomes after Stages 1.5+1.6, indicating a third mutation source.

Candidate sources (diagnostic must distinguish):
- **event_bus subscriber list mutations** — handlers register/deregister mid-game and the registry persists across games
- **Module-level state** — module globals not reset between games (typical: cached lookup tables, "lazy init" objects that retain state)
- **Random state contamination** — secondary RNG that doesn't get re-seeded
- **Cached database connections / cursors** — stateful connections that survive across games
- **File handles / log buffers** — accumulated state in I/O paths

## Deliverables

### D1: Hash logging instrumentation

Add temporary instrumentation that snapshots candidate mutable state at game start and game end. Output to a structured log file for diff-ability.

**File:** `engine/_diagnostic/state_hash_logger.py` (use the `_diagnostic/` convention if `_research/` doesn't fit; or `_research/` if that's already established as the pattern). New file.

**Functions:**

```python
def snapshot_state(label: str, gs=None) -> dict:
    """Return a dict of {component_name: hash} for all candidate mutable state.
    
    Components to hash:
    - event_bus subscriber dict (sorted keys, hash of each handler's qualname)
    - Every module-level dict / list in engine/ (use a registry)
    - Card class-level state (if any persists)
    - Active database connection state (if applicable)
    - Random state (random.getstate(), numpy if used)
    """
    
def log_snapshot(label: str, snapshot: dict, log_path: Path):
    """Append snapshot to log file as JSON line."""
```

The snapshot function must be **comprehensive over candidate sources** — better to hash too many things and find that 4 of 5 are stable than miss the actual mutation source.

### D2: Diagnostic run

Run 5 sequential games with seed=42 against a deterministic matchup (BE mirror is the simplest). Snapshot state at start and end of each game.

```python
# scripts/run_determinism_diagnostic.py (or similar; new file)
for game_n in range(5):
    snap_start = snapshot_state(f"game_{game_n}_start", gs)
    log_snapshot(...)
    
    play_one_game(seed=42, deck_a="boros_energy_modern", deck_b="boros_energy_modern")
    
    snap_end = snapshot_state(f"game_{game_n}_end", gs)
    log_snapshot(...)
    
    # Game outcome should be identical across all 5 iterations if determinism holds
    log_outcome(game_n, ...)
```

Run the diagnostic. Outputs:
1. The 10 state snapshots (5 starts + 5 ends)
2. The 5 game outcomes

### D3: Diff analysis

For each component in the snapshot:
- If `start_n` matches `start_n+1` for all n, that component is stable across game boundaries → not the bug source
- If `start_n` differs from `start_n+1`, that component carries state across games → candidate bug source
- If outcomes differ across 5 games but ALL components hash identically at game starts → the bug is in something the snapshot didn't capture (expand snapshot, re-run)

Produce a diff table:

| Component | start_0 | start_1 | start_2 | start_3 | start_4 | Stable? |
|---|---|---|---|---|---|---|
| event_bus | h0 | h1 | h2 | ... | ... | NO (bug candidate) |
| ... | ... | ... | ... | ... | ... | YES |

### D4: Targeted fix

Based on D3 findings, fix the divergent component. Implementation depends on what's found:
- **Subscriber leak:** explicit `event_bus.reset()` at game start
- **Module-level state:** find and reset the offending module global; ideally refactor to per-game state
- **RNG leak:** explicit re-seed of all RNGs at game start
- **Connection leak:** scope connections to per-game lifecycle

Spec cannot prescribe the fix without seeing the diagnostic output. Mid-execution amendment will document the actual fix path.

### D5: Validation

After fix:
1. Re-run D2 diagnostic. All component hashes at `start_n` should be identical across n (or, allowed to differ in known/expected ways with documented justification).
2. All 5 game outcomes identical.
3. Run BE mirror at n=1000 seed=42 twice in sequence. Both runs return identical aggregate (e.g., both 49.4% to the percentage point, or both 50.1%, etc. — the actual value doesn't matter; what matters is that two runs return the SAME value).
4. Run a 14-deck gauntlet at n=200 seed=42 twice. All 14 matchup results identical between runs.

### D6: Remove or finalize instrumentation

The state_hash_logger.py should NOT live in production code. Two options:
- **Remove entirely** after the bug is found, with the diagnostic script kept in `engine/_research/` for future re-use
- **Keep as `_diagnostic/` opt-in** for any future determinism investigation, gated behind an env var so it's a no-op in normal runs

Decide based on whether the instrumentation has reusable value. Default: keep in `_research/` with a README documenting "run this if determinism breaks again."

## Validation gates

**Gate 1:** D2 diagnostic completes and produces a diff table identifying ≥1 bug-candidate component (if it produces zero candidates and outcomes still differ, snapshot was insufficient — expand and re-run).
**Gate 2:** D4 fix targets the identified component and changes nothing else.
**Gate 3:** D5 validation shows determinism holds: 2 sequential gauntlet runs at n=200 produce identical per-matchup results.
**Gate 4:** D5 BE mirror at n=1000 seed=42 produces identical aggregate across 2 sequential runs.
**Gate 5:** D6 instrumentation either removed from production paths or gated behind opt-in flag.
**Gate 6:** Drift-detect runs clean.

## Stop conditions

**Ship when:** All 6 gates pass.

**Stop and amend if:**
- D3 diff reveals MULTIPLE bug-candidate components (not just one) — fix one at a time, validate independently, document each in amendments
- The fix requires touching more than 3 files outside the diagnostic surface — likely scope creep, re-spec
- D5 validation reveals determinism holds at n=200 but breaks at n=1000 — there's a frequency-dependent component (state that only mutates after many games), expand instrumentation to hash periodically during long runs

**DO NOT:**
- Do NOT bundle within-matchup parallelism implementation into this spec. That's downstream; ship determinism first, parallelism second.
- Do NOT touch any APL files. Bug is in engine/.
- Do NOT optimize anything during the fix. Targeted minimal change only.

## Why this order

Diagnostic-first because the cost of instrumentation is small and the cost of guessing wrong about which component is the source is large. We've already paid the "guess and fix" cost twice (Stages 1.5 + 1.6); a third guess-and-check is bad allocation. Cheap instrumentation reveals the answer in 5 minutes of run time.

D6 instrumentation cleanup is last because we don't know yet whether the diagnostic has reuse value until D4 reveals what we found.

- **Determinism regression test.** Add a test to `tests/` that runs BE mirror twice at small n and asserts outcomes match. Catches future determinism breaks at PR time. Estimated 15 min, can fold into D6 if quick.
