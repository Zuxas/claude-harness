# Phase 3.5 Stage B -- combat modifiers

## Goal

Wire 4 combat-modifier keyword effects into match-runner. Closes the
HASTE bug (Ragavan can't attack T1 in match mode), adds DEFENDER
attacker filter, activates Stage A's blocker-side LIFELINK placeholder
(threading through return tuple), and adds minimal tap-state model
for VIGILANCE.

## Scope

### In scope

1. **HASTE filter fix.** `_resolve_combat` attacker filter currently
   `not getattr(c, 'summoning_sickness', False)` doesn't OR with
   `KWTag.HASTE`. Fix both attacker selection paths.

2. **DEFENDER filter.** Add `KWTag.DEFENDER not in c.tags` to
   attacker filter. Zero defenders in BE matchups; correctness fix.

3. **Blocker-side LIFELINK activation.** Stage A's `pass` placeholder
   in Pass 1 of `_resolve_strike_step` activates as
   `defender_lifelink_gain += blk_pwr` per lifelink blocker. Thread
   through `_resolve_combat` return tuple as 5th value. Update both
   callers (`_run_player_turn`, `_run_match_with_combo`).

4. **VIGILANCE tap-state.** Add `tapped_from_attack` flag (Card
   attribute, set/cleared in match-runner). Filter blockers via
   `not getattr(c, 'tapped_from_attack', False)`. Set after attack
   unless `KWTag.VIGILANCE in c.tags`. Clear at start of each
   player's turn.

### Explicitly out of scope

- Stage A's menace/evasion coverage -- already shipped in `99037a9`
- Protection keywords (hexproof/ward/protection/shroud) -- Stage C
- FLASH priority window -- Stage D
- Death replacement (undying/persist) -- Stage E
- Tap state for any purpose other than vigilance-blocking -- e.g.
  no tap-for-mana modeling. Keep tap-state minimal and combat-only.

## Mid-execution amendments

### Amendment 4: B.4 stop-condition trigger explained as correct behavior (2026-04-27)

**Trigger:** B.4 canonical 1k Modern landed 65.3% (was 65.8%, -0.5pp).
Spec predicted UPWARD shift +0.5 to +2pp from haste fix benefiting BE
Ragavan T1 attacks. Direction-of-shift stop condition fired (acceptance
band 65-69% technically holds at 65.3% but "lands DOWN instead of up"
trigger fires).

**Investigation:** Mono Red matchup shifted -6.5pp (52.8% -> 46.3%),
which dominates the aggregate change at 4.1% field share. Other
sim-source matchups within +/-1.5pp.

**Root cause:** spec prediction was incomplete. HASTE fix is symmetric
-- both decks gain T1 attack capability for haste creatures. Mechanical
verification (pre-commit):

| Deck | HASTE creature count | Names |
|---|---|---|
| BE canonical (.txt) | 4 | 4x Ragavan |
| Mono Red Modern | **8** | 4x Goblin Guide + 4x Monastery Swiftspear |

Mono Red has 2x BE's haste density. Pre-Stage-B Mono Red was over-rated
for BE because Mono Red's T1 hastes (Goblin Guide, Swiftspear) couldn't
attack T1 either. Post-Stage-B, the matchup is more accurate; Mono Red
gains relatively more from the symmetric fix proportional to its
haste-creature density.

**Decision:** the shift is honest model improvement, not regression.
Direction-of-shift stop condition was based on incomplete prediction
model that only counted BE's haste creatures. Generalizable lesson
for future stage specs: predict shift magnitude AND direction PER
MATCHUP, not aggregate, when the new keyword effect is symmetric
across decks with asymmetric keyword density. Also account for SIM-vs-DB
source -- only sim-source matchups can shift.

**Variant edge:** Variant +12.0pp -> +12.9pp (+0.9pp GREW). Variant
gained on sim-source matchups (Eldrazi Ramp +13pp via SIM path)
where canonical ate the Mono Red hit. Variant edge growth is the
clean signal that variant's aggro structure benefits more from the
fix than canonical's value-engine structure.

**B.6 spot check resolution:** canonical Eldrazi Ramp held at 49.2%
because that matchup is [DB]-cached, not sim-source. Engine changes
can't shift DB lookups. Variant Eldrazi Ramp at 91.9% [SIM] confirms
engine working -- variant gained on this matchup post-Stage-B because
variant runs through the engine path while canonical pulls from DB
cache.

### Amendment 5: tagger-fix activation reveals partial-effect at SHIP (2026-04-28)

**Trigger:** Tagger-load-path-unification spec (commit 199d28e) shipped.
Stage B's HASTE/DEFENDER/VIGILANCE/blocker-lifelink filters were silent
against the 3 stub-loaded opps (Izzet Prowess, Domain Zoo, Esper

above, -6.5pp BE-vs-Mono-Red) WAS captured correctly because Mono
Red is .txt-loaded; the stub-loaded subset's haste-creature impact
was missed.

**Empirical activation, post-tagger-fix:** The single largest
canonical shift attributable to Stage B's HASTE filter activating
on stub-loaded matchups is Izzet Prowess canonical 49.4% → 41.3%
(Δ -8.1pp), driven by opp's Monastery Swiftspear / Slickshot Show-Off
gaining T1 attack capability. This is a Stage B effect that
materialized only after the tagger-fix landed, not a Stage B
regression. The shift aligns with real-world tournament data showing
Izzet Prowess slightly favored vs Boros Energy.

Subset-aggregate canonical shift on stub-loaded matchups: -4.52pp
(Stage A MENACE + Stage B HASTE/etc. + Stage C HEXPROOF combined,
but HASTE on Izzet Prowess is the dominant single contributor).

The variant edge documented at SHIP (+12.9pp) was partial-effect.
Full-effect post-tagger-fix variant edge: ~+14.2pp.
