## Background

### Magic Comprehensive Rules primer (verify CR section numbers in implementation)

**Hexproof (CR 702.11ish):** "This permanent can't be the target of spells or abilities your opponents control." Asymmetric -- the controller can still target with their own spells/abilities. Common in Modern: Slippery Bogle / Bogle decks; Sythis Enchantress; Geist of Saint Traft; some sideboard creatures.

**Shroud (CR 702.18ish):** "This permanent can't be the target of spells or abilities." Stronger than hexproof -- not even the controller can target. Rare in modern Modern; mostly older creatures (Troll Ascetic). Implementation must handle, but expect zero or near-zero impact in actual matchups.

**Ward N (CR 702.21ish):** "Whenever this permanent becomes the target of a spell or ability an opponent controls, counter that spell or ability unless that player pays N." (Note: ward variants exist with non-mana costs, e.g., Ward -- Pay 2 life, or Ward -- Discard a card. Spec assumes mana-cost ward only for v1; non-mana ward is a follow-up.) Common in modern Modern: Stoneforge Mystic-style creatures, Sheoldred-style cards, various recent print runs.

**Protection from <X> (CR 702.16ish):** "DEBT" mnemonic -- target/damage/equip-attach/block all restricted with respect to source X. Stage C handles **T (target) only**. The other letters (D, E, B) are deferred:
- D (damage prevention): Stage A already handles some via the block-eligibility framework; full damage prevention deferred to follow-up
- E (equip/attach): N/A for most Modern decks; Auras attaching is rare; deferred
- B (block): Stage A's block-eligibility already handles part of this for menace/flying/reach; full DEBT-block is a follow-up
- T (target): THIS SPEC

### Why Stage C is highest-risk

**Surface area:** Every removal spell, every targeted ability, every aura/equipment attach. Estimated 50-100 distinct target-selection sites across the engine + ~15 APLs.

**Asymmetric impact (Stage B Amendment 4 lesson):** Hexproof/ward/protection density varies wildly across decks. BE has near-zero hexproof creatures; some opponent decks have multiple. Predicting aggregate shift without per-matchup density check is exactly the failure mode Stage B caught. The keyword-density check is REQUIRED, not optional.

**Multi-keyword interaction:** A creature with hexproof AND ward isn't "double-protected" -- hexproof preempts the ward trigger because the spell can't even be cast. A creature with protection from red AND ward 2 against a red removal spell is protected by the protection (no targeting), not the ward. Implementation order matters; tests must cover the interaction.

**Cost integration:** Ward forces an additional payment or spell counter. APL needs to know "can I afford the ward tax?" before committing to targeting. This is the trickiest part of Stage C and likely produces the most amendments.

**Functions to implement:**

```python
def can_be_targeted_by(target: Card, source: Card | Player, controller: Player) -> bool:
    """High-level: does targeting succeed given source/controller relationship?
    
    Returns False if:
    - target has SHROUD (no one can target)
    - target has HEXPROOF and controller != target.controller
    - target has PROTECTION_<X> and source matches X
    
    Note: ward does NOT make can_be_targeted_by return False -- ward triggers
    AFTER targeting is declared, on resolution. Ward cost handling is a
    separate function (see ward_cost below).
    """

def ward_cost(target: Card, source: Card | Player, controller: Player) -> Optional[Cost]:
    """Returns ward Cost if target has WARD and controller is opponent of target.controller.
    Returns None if no ward applies.
    
    Caller (APL) decides whether to pay the cost or abandon targeting.
    """

def is_protected_from(target: Card, source: Card | Player) -> bool:
    """Convenience wrapper: True if PROTECTION_<X> applies to source.
    
    'X' can be: a color, a card type, a subtype, 'multicolored', 'monocolored',
    'everything', or other variants. Implementation must handle at minimum:
    - Protection from <color> (5 colors)
    - Protection from creatures
    - Protection from multicolored
    
    Other protection variants (from spells, from instants, from a specific name)
    can return False with a TODO comment; they're rare in Modern and can be
    added per-card as needed.
    """
```

## Risk register

**R1: Targeting code chokepoint vs spread (high impact, medium probability).**
If targeting happens through one helper in `engine/game_state.py`, integration is trivial -- patch the helper, all sites benefit. If targeting is duplicated across many sites, C.3 budget blows out. Mitigation: identify chokepoint in C.1 pre-flight; if no chokepoint, scope-down to BE + top-5 field-share opponents and defer the rest. **Status per A1:** chokepoint EXISTS at `engine/card_handlers_verified.py:_damage_any_helper` (38 callers); R1 resolves to good case.

**R2: Ward cost cascade through APL resource accounting (high impact, medium probability).**
BE plays Lightning Helix on a Ward 2 creature: that's now 1R1 + 2 = 1R3 for Helix. Does BE's mana check account for this dynamically? If not, BE will commit to Helix and then fail to cast it, leaving energy/mana spent in a bad state. Mitigation: write the synthetic test for ward-cost-too-expensive case as part of C.1 tests; verify BE handles "too expensive, drop target" gracefully before integrating.

**R3: Multi-keyword interaction edge cases (medium impact, low probability).**
Hexproof + Ward, Protection + Ward, Protection from creatures + Lightning Helix (which is an instant, not a creature -- so protection from creatures DOESN'T block it). Mitigation: explicit synthetic tests for the interaction matrix in C.1; comment in code where the interaction order matters.

**R4: Stage A/B regression introduced by Stage C (low impact, medium probability).**
Stage A handled block-eligibility; Stage B handled combat modifiers. If Stage C touches `engine/game_state.py:_do_combat` or related, it could regress. Mitigation: run Stage A and Stage B's synthetic tests as part of Gate 2 (`pytest tests/test_block_eligibility.py tests/test_combat_modifiers.py` if those exist).

**R5: Prediction model wrong for an opponent we don't fully understand (medium impact, low probability).**
Some Modern decks have keyword interactions specific to their archetype that the spec author doesn't know. Mitigation: per-matchup density check forces explicit reasoning; if reasoning is unclear, mark prediction as "low-confidence" and accept wider tolerance for that matchup.

**R6: Stale baseline anchoring causes false-fail or false-pass on C.4 (medium impact, high probability without mitigation).**
Engine code changed between Stage B SHIP (c95ea55, 2026-04-27 12:00) and Stage C re-execution (2026-04-28). Specifically, Guide attack-trigger fix at cf75e1a landed in that window. Cache-fix Gate 2 confirmed +0.3-0.7pp variant drift attributable to engine evolution. Mitigation per A4: capture C.0 fresh baseline at HEAD before C.1 begins; predictions and acceptance bands anchor on C.0 numbers, not stage-doc numbers.

## Why this order

- **Baseline capture (C.0) before any code change (per A4):** if engine drifted between Stage B and now, predictions anchored on Stage B numbers will mis-fire. Fresh baseline at HEAD eliminates the stale-anchor risk class.
- **Helpers before integration (C.1 before C.2/C.3):** broken helpers contaminate every integration point. Fix-once at the helper layer, then integrate confidently.
- **BE before opponents (C.2 before C.3):** BE is the deck under primary study. Verifying BE's targeting works before auditing ~15 opponent APLs means we're not chasing two problems at once if validation breaks.
- **Synthetic tests before live validation:** synthetic tests isolate the keyword logic from the rest of the simulator. Live gauntlet failure could be Stage C, could be Stage A/B regression, could be prediction error -- synthetic tests narrow the search space.
- **Pre-prediction before validation (C.4 prereq):** per the falsifiable-prediction-model lesson. If you can't write predictions, you can't interpret results. Writing them as a separate step means the prediction is independent of the result-bias of having seen the actuals.
